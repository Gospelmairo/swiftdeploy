# SwiftDeploy

A declarative CLI tool that generates and manages a full container stack from a single `manifest.yaml` file.

**Author:** Mairo Gospel ([@Gospelmairo](https://github.com/Gospelmairo))

---

## Overview

SwiftDeploy reads `manifest.yaml` as the single source of truth and generates all config files automatically. Edit the manifest — SwiftDeploy handles the rest.

---

## Project Structure

```
swiftdeploy/
├── manifest.yaml              # Single source of truth — edit only this
├── swiftdeploy                # CLI tool (executable)
├── Dockerfile                 # App image build
├── app/
│   ├── main.py                # Flask API service
│   └── requirements.txt
├── templates/
│   ├── nginx.conf.j2          # Nginx config template
│   └── docker-compose.yml.j2  # Compose template
└── README.md
```

Generated files (do not edit manually):
- `nginx.conf` — generated by `swiftdeploy init`
- `docker-compose.yml` — generated by `swiftdeploy init`

## Setup — Fresh Machine to Running Stack

### Prerequisites

- Docker Desktop v24+
- Python 3.11+
- pip

### 1. Clone the repo

```bash
git clone https://github.com/Gospelmairo/swiftdeploy.git
cd swiftdeploy
```

### 2. Install CLI dependencies

```bash
pip install pyyaml jinja2
```

### 3. Build the app image

```bash
docker build -t swift-deploy-1-node:latest .
```

### 4. Deploy

```bash
./swiftdeploy deploy
```

## Subcommand Walkthrough

### `init`
Parses `manifest.yaml` and generates `nginx.conf` and `docker-compose.yml`:
```bash
./swiftdeploy init
```

### `validate`
Runs 5 pre-flight checks before deploying:
```bash
./swiftdeploy validate
```
Checks:
1. `manifest.yaml` exists and is valid YAML
2. All required fields are present and non-empty
3. Docker image exists locally
4. Nginx port is not already bound
5. Generated `nginx.conf` syntax is valid

### `deploy`
Runs `init`, brings the stack up, and waits for health checks to pass (60s timeout):
```bash
./swiftdeploy deploy
```

### `promote`
Switches deployment mode between `stable` and `canary`:
```bash
./swiftdeploy promote canary
./swiftdeploy promote stable
```
- Updates `mode` in `manifest.yaml`
- Regenerates `docker-compose.yml`
- Restarts only the app container
- Confirms new mode via `/healthz`

### `teardown`
Removes all containers, networks, and volumes:
```bash
./swiftdeploy teardown
./swiftdeploy teardown --clean   # also deletes generated config files
```

## API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| GET | `/` | Welcome message with mode, version, timestamp |
| GET | `/healthz` | Health check — returns status and uptime |
| POST | `/chaos` | Simulate degraded behaviour (canary mode only) |

### Chaos Modes (canary only)

```bash
# Slow responses
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "slow", "duration": 3}'

# Random errors
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "error", "rate": 0.5}'

# Recover
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "recover"}'
```

## manifest.yaml Reference

```yaml
services:
  image: swift-deploy-1-node:latest   # Docker image name
  port: 3000                          # App container port
  mode: stable                        # stable or canary
  version: "1.0.0"                    # App version
  restart_policy: unless-stopped      # Docker restart policy
  proxy_timeout: 30                   # Nginx proxy timeout in seconds

nginx:
  image: nginx:latest                 # Nginx image
  port: 8080                          # Host port exposed to internet

network:
  name: swiftdeploy-net               # Docker network name
  driver_type: bridge                 # Network driver
```
